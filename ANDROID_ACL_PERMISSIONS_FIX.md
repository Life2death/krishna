## The blocker
Invoking the command from the running WebView returns:
```
device-control.setTorch not allowed. Plugin not found
```
This is the **Tauri v2 ACL** rejecting the call. Two root causes, both must be fixed:

1. **No capability targets Android.** All three files in `src-tauri/capabilities/` declare `"platforms"` of only `macOS` (`default.json`), or `windows`/`linux` (`cross-platform.json`, `presence.json`). There is **no** capability for `android`, so the main window on the phone gets **zero** ACL grants.
2. **The `device-control` plugin ships no permissions.** It's an **inline** plugin (`tauri::plugin::Builder::new("device-control")` in `src-tauri/src/device_control.rs`, in the app crate — not a separate plugin crate). It's absent from `src-tauri/gen/schemas/acl-manifests.json`, has no `permissions/` dir, and `build.rs` doesn't generate any. Inline plugins do **not** get ACL permissions auto-generated, so there is no `device-control:allow-set-torch` permission to grant.

The 7 commands (Kotlin `@Command` names → Tauri command names): `setTorch`→`set_torch`, `listApps`→`list_apps`, `launchApp`→`launch_app`, `openSetting`→`open_setting`, `setVolume`→`set_volume`, `setDnd`→`set_dnd`, `requestBluetoothEnable`→`request_bluetooth_enable`. (Confirm the snake_case mapping Tauri expects — it lowercases/snake-cases command idents.)

## Fix — verify each step against current Tauri v2 docs ("Plugin Permissions", "Capabilities") before committing; do not guess at ACL internals

**Part A — generate ACL permissions for the plugin.** Recommended canonical path is to give the plugin a real permission manifest. Evaluate, in order of preference:

- **Option 1 (cleanest, supported): extract `device-control` into a local plugin crate** under e.g. `src-tauri/plugins/device-control/` as a path dependency, with its own `build.rs` calling `tauri_plugin::Builder::new(&COMMANDS).build()` and a `permissions/` dir. A real plugin crate gets `allow-*` permissions + an autogenerated `default` set keyed to the plugin name, registered into the app's ACL automatically. Move `device_control.rs` + the Kotlin `@Command` registration accordingly.
- **Option 2 (less restructuring): keep inline, hand-author the permission set.** Add build-dep `tauri-plugin = "2"`, and in `build.rs` (before `tauri_build::build()`) generate permissions for the command list, **ensuring the generated namespace is `device-control`** (this is the tricky part for an inline plugin — verify the generated `acl-manifests.json` actually contains `device-control:allow-set-torch` etc., not the app namespace). If the namespace can't be forced to `device-control`, fall back to Option 1.

After this, `src-tauri/gen/schemas/acl-manifests.json` must contain `device-control` permissions (rebuild, then grep it to confirm).

**Part B — grant the permissions on Android.** The main window needs a capability that (a) targets `android` and (b) lists the device-control permissions. Either add `"android"` to an existing cross-platform capability's `platforms` and append the perms, or create `src-tauri/capabilities/android.json`:
```json
{
  "$schema": "../gen/schemas/mobile-schema.json",
  "identifier": "android",
  "description": "Main-window capability on Android — core IPC + device-control plugin.",
  "windows": ["main"],
  "platforms": ["android"],
  "permissions": [
    "core:default",
    "core:event:default",
    "device-control:allow-set-torch",
    "device-control:allow-list-apps",
    "device-control:allow-launch-app",
    "device-control:allow-open-setting",
    "device-control:allow-set-volume",
    "device-control:allow-set-dnd",
    "device-control:allow-request-bluetooth-enable"
  ]
}
```
(Use `mobile-schema.json` / `android-schema.json` — both exist under `gen/schemas/`. Confirm the exact permission identifier strings against the regenerated ACL manifest; they may be `device-control:default` if a default set is generated.) Note core HTTP/window perms the desktop capabilities grant — decide which the Android main window also needs (it currently has none).

## FOLLOW-UP (found by on-device test, 2026-06-28) — command-name casing mismatch

The ACL work landed correctly: after rebuild, the live invoke error changed from `Plugin not found` → **`Command not found`** (the plugin + permissions are now recognized). But `setTorch` still fails because of a naming mismatch surfaced only by a real invoke (not `cargo check`):

- Frontend `invoke` (`src/lib/actions.ts`): **camelCase** — `setTorch`, `listApps`, `launchApp`, `openSetting`, `setVolume`, `setDnd`, `requestBluetoothEnable`.
- Kotlin `@Command` methods (`DeviceControlPlugin.kt`): **camelCase** — same.
- `build.rs` ACL `commands(&[...])`: **snake_case** — `set_torch`, `list_apps`, … → ACL only knows snake_case, so an invoke of `setTorch` is "Command not found".

**Fix — make everything snake_case** (Tauri convention; keeps the already-correct `allow-set-torch` permission IDs and `android.json` untouched):
1. `src/lib/actions.ts` — change the 6 invoke strings: `plugin:device-control|setTorch` → `|set_torch`, `launchApp`→`launch_app`, `openSetting`→`open_setting`, `setVolume`→`set_volume`, `setDnd`→`set_dnd`, `requestBluetoothEnable`→`request_bluetooth_enable` (and `listApps`→`list_apps` if present).
2. `DeviceControlPlugin.kt` — rename the 7 `@Command` methods to snake_case (`fun set_torch(invoke: Invoke)`, …). Tauri's Android dispatch matches the invoked command string to the method name, so the method names must equal the snake_case command.
3. Leave `build.rs` (snake_case) and `capabilities/android.json` (`allow-set-torch`) **as-is** — they're already right.
4. **Do NOT** instead change `build.rs` to camelCase — Tauri would regenerate the permission IDs (likely `allow-setTorch`) and break `android.json`.

**Re-verify on device** (no full rebuild of JS needed if dev server is up, but ACL/Kotlin changes need a Gradle rebuild): use `cdp-torch.mjs` (in worktree) →
`window.__TAURI_INTERNALS__.invoke('plugin:device-control|set_torch',{on:true})` should return OK and the flashlight should physically turn on. (`window.__TAURI__` is off; use `__TAURI_INTERNALS__`.) Then `set_torch{on:false}`, then `list_apps`, etc.
